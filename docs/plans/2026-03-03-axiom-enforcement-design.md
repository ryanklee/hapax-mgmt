# Axiom Enforcement Design: Axiom-as-Law

**Date**: 2026-03-03
**Status**: Approved
**Approach**: C (Axiom-as-Law / Precedent-First)

## Summary

Full 7-layer axiom enforcement architecture for the hapaxromana system.
Axioms gain enforceable meaning through accumulated precedent (case law),
not solely through upfront rule derivation. Starts with 2 axioms
(single_user, executive_function), infrastructure designed to scale.

**Split**: Definitions in hapaxromana, enforcement code in ai-agents.

## Threat Model

Both LLM agent drift (agents making decisions that violate axioms) and
system design drift (codebase accumulating axiom-inconsistent decisions).
Enforcement targets AI behavior at decision time and the codebase over time.

## Architecture Overview

```
hapaxromana/axioms/          shared/
  registry.yaml               axiom_registry.py     (load/validate registry)
  implications/                axiom_precedents.py   (Qdrant precedent store)
    single-user.yaml           axiom_derivation.py   (one-shot derivation pipeline)
    executive-function.yaml    axiom_tools.py        (agent tools: check + record)
  precedents/seed/
    single-user-seeds.yaml   agents/
    executive-function-      health_monitor.py       (new axiom check group)
      seeds.yaml             drift_detector.py       (axiom compliance pass)
                             briefing.py             (axiom section)

                             cockpit/
                               chat_agent.py         (review tools)
```

## Layer Mapping to Existing System

| Layer | What Exists | What's New |
|-------|------------|-----------|
| L1: Static Injection | CLAUDE.md, rules/, operator.json axiom booleans | Richer axiom text in agent prompts (from registry) |
| L2: Concrete Rules | 49+ health checks, service tiers, nudges | New `axioms` check group (4 checks) |
| L3: LLM Decomposition | Context tools (constraints/patterns) | Derivation pipeline (one-shot seeding) |
| L4: Top-Down Audit | Drift detector (docs vs reality) | Axiom compliance pass in drift detector |
| L5: Decision-Time | Axioms injected as booleans | `check_axiom_compliance` tool on all LLM agents |
| L6: Operator Authority | Profile corrections, decision capture | `/axioms review`, precedent confirm/reject |
| L7: Precedent Database | Behavioral data (traces, shell, git) | `axiom-precedents` Qdrant collection |

## Component Designs

### 1. Axiom Registry (hapaxromana)

**Directory**: `hapaxromana/axioms/`

**`registry.yaml`**:
```yaml
version: 1
axioms:
  - id: single_user
    text: >
      This system is developed for a single user and by that single user,
      the operator (Hapax). This will always be the case. All decisions
      must be made respecting and leveraging that fact.
    weight: 100
    type: hardcoded
    created: 2026-03-03
    status: active
    supersedes: null

  - id: executive_function
    text: >
      This system serves as externalized executive function infrastructure.
      The system is designed with cognitive load awareness - task initiation, sustained attention,
      and routine maintenance are genuine cognitive challenges. The system
      must compensate for these, not add to cognitive load.
    weight: 95
    type: hardcoded
    created: 2026-03-03
    status: active
    supersedes: null
```

Schema properties per axiom:
- `id`: snake_case identifier (stable, used as FK everywhere)
- `text`: Full plain-language axiom text
- `weight`: 0-100 abstract weight (operator-declared)
- `type`: `hardcoded` (never violated) or `softcoded` (defeasible when outweighed)
- `created`: ISO date
- `status`: `active` or `retired`
- `supersedes`: ID of axiom this replaces (null if original)

**`implications/<axiom-id>.yaml`**:
```yaml
axiom_id: single_user
derived_at: 2026-03-03
model: claude-sonnet
derivation_version: 1
implications:
  - id: su-arch-001
    tier: T0
    text: "No multi-user authentication or authorization system"
    enforcement: block
    canon: textualist
  - id: su-arch-002
    tier: T0
    text: "No multi-tenant data isolation patterns"
    enforcement: block
    canon: textualist
  - id: su-design-001
    tier: T1
    text: "No user-switching UI or session management"
    enforcement: review
    canon: purposivist
  # ... generated by derivation pipeline, reviewed by operator
```

Implication properties:
- `id`: `<axiom-prefix>-<category>-<seq>` (e.g., su-arch-001)
- `tier`: T0 (existential/block), T1 (significant/review), T2 (minor/warn), T3 (cosmetic/lint)
- `text`: Concrete implication in plain language
- `enforcement`: `block` | `review` | `warn` | `lint`
- `canon`: Which interpretive strategy produced this (textualist, purposivist, absurdity, omitted-case)

**`precedents/seed/<axiom-id>-seeds.yaml`**:
```yaml
axiom_id: single_user
precedents:
  - id: sp-su-001
    situation: "Mobile access via Tailscale to home services"
    decision: compliant
    reasoning: >
      Multi-device access for the single user is consistent with the axiom.
      Tailscale authenticates the device, not a separate user. No multi-user
      scaffolding is introduced.
    tier: T1
    distinguishing_facts:
      - "Single user accessing from multiple personal devices"
      - "No new authentication layer for user identity"
    created: 2026-03-03
    authority: operator
```

### 3. Derivation Pipeline (ai-agents)

**New module**: `shared/axiom_derivation.py`
**CLI**: `uv run python -m shared.axiom_derivation --axiom single_user`

One-shot process, operator-triggered only:

1. Load axiom text from registry
2. Gather codebase context:
   - File tree of ai-agents/  (via glob)
   - Key module docstrings (health_monitor, chat_agent, etc.)
   - Current CLAUDE.md content
3. Construct derivation prompt with interpretive canons:
   - Textualist: What does the axiom literally say?
   - Purposivist: What is it trying to achieve?
   - Absurdity doctrine: Filter results no reasonable person would endorse
   - Omitted-case canon: Don't add things the axiom doesn't state
4. Run self-consistency: N=3 derivations (model: balanced)
5. Majority vote on each implication, resolve tier by consensus
6. Output YAML to stdout (or `--output` to write file)
7. **Operator reviews and edits before committing to hapaxromana**

Also generates initial seed precedents from implications:
- Each T0/T1 implication produces a corresponding seed precedent
- These are starting points, not final authority

**Re-derivation triggers** (all manual):
- Axiom text changes
- Major codebase evolution (new agent, new infra component)
- Operator requests

### 6. Existing System Modifications

#### 6a. Drift Detector Axiom Pass

**Modify**: `agents/drift_detector.py`

Add second pass after existing docs-vs-reality comparison:

1. Load active axioms and T0/T1 implications
2. For each implication with `enforcement: block` or `enforcement: review`:
   - LLM evaluates: does the current codebase violate this implication?
   - Uses infrastructure manifest + file tree as evidence
3. Report violations as drift findings with category `axiom-violation`
4. T0 violations: severity=high. T1 violations: severity=medium.

Runs on existing weekly schedule (Sunday 03:00). No new timer.

#### 6b. Health Monitor Axiom Checks

**Modify**: `agents/health_monitor.py`

New `@check_group("axioms")` with 4 deterministic checks:
- `axiom.registry_exists`: Registry file accessible and parseable
- `axiom.precedent_collection`: Qdrant `axiom-precedents` collection exists
- `axiom.implications_current`: Implication files exist for all active axioms
- `axiom.pending_review`: Count of agent precedents awaiting review (warn > 10)

Tier: OBSERVABILITY (default in `shared/service_tiers.py`).

#### 6c. Briefing Axiom Section

**Modify**: `agents/briefing.py`

Add to briefing data collection (before LLM synthesis):
- Count of new agent precedents since last briefing
- Any T0/T1 findings from most recent drift detector axiom pass
- Pending review count

Appears in briefing only when there's something to report.

#### 6d. Cockpit Axiom Management

**Modify**: `cockpit/chat_agent.py`

New tools:
- `review_pending_precedents()`: Show agent-created precedents awaiting review
- `confirm_precedent(precedent_id)`: Promote from agent to operator authority
- `reject_precedent(precedent_id, correction)`: Supersede with operator correction

New slash command registered in cockpit:
- `/axioms`: List active axioms with stats
- `/axioms review`: Show pending agent precedents
- `/axioms search <query>`: Search precedent database

#### 6e. Operator.json Evolution

**Modify**: `profiles/operator.json`

Current axiom booleans (`single_user: true`, `executive_function: true`) remain
for backward compatibility. `get_system_prompt_fragment()` in `shared/operator.py`
enhanced to load full axiom text from registry when available, falling back to
boolean flags if registry is missing.

## Data Flow

### Decision-Time (Real-Time, Per-Agent-Call)

```
Agent encounters decision
  -> check_axiom_compliance(situation)
    -> PrecedentStore.search(axiom_id, situation)
      -> Qdrant vector search (filtered by axiom_id)
        -> Return similar precedents
    -> If no matches: load axiom text + implications
  -> Agent reasons with precedents/axioms
  -> Agent makes decision
  -> record_axiom_decision(...)
    -> PrecedentStore.record(authority="agent")
      -> Embed + upsert to Qdrant
```

### Periodic Audit (Weekly, Drift Detector)

```
drift_detector.py runs (Sunday 03:00)
  -> Existing: docs vs reality comparison
  -> New: axiom compliance pass
    -> Load axioms + T0/T1 implications
    -> LLM evaluates each against codebase
    -> Report violations as drift findings
  -> Notify via shared.notify if violations found
```

### Operator Review (On-Demand, Cockpit)

```
Operator sees "N pending precedents" in briefing
  -> /axioms review (in cockpit)
    -> PrecedentStore.get_pending_review()
    -> Display each with situation, decision, reasoning
  -> Operator confirms or rejects each
    -> confirm: promote authority to "operator"
    -> reject: supersede with correction (authority="operator")
```

### Seeding (One-Time, Derivation Pipeline)

```
Operator runs: uv run python -m shared.axiom_derivation --axiom single_user
  -> Load axiom text
  -> Gather codebase context
  -> LLM derivation (N=3 self-consistency)
  -> Output implications YAML
  -> Operator reviews, edits, commits to hapaxromana
  -> PrecedentStore.load_seeds() ingests seed precedents
```

## Success Criteria

1. Two axioms defined in registry, implications derived and committed
2. Seed precedents loaded into Qdrant `axiom-precedents` collection
3. LLM agents can call `check_axiom_compliance` and get relevant precedents
4. Agents record decisions that become searchable precedents
5. Drift detector reports axiom violations alongside docs drift
6. Operator can review/confirm/reject agent precedents via cockpit
7. Health monitor verifies axiom infrastructure is operational
8. All existing tests continue to pass

## Proportionality Budget

- Decision-time tool calls: < 2s each (Qdrant search + format)
- Derivation pipeline: minutes per run (infrequent, manual trigger)
- Drift detector axiom pass: adds ~30s to existing weekly run
- Health monitor axiom checks: < 5s (HTTP calls to Qdrant)
- Total overhead on normal agent operation: negligible (tool call only when agent decides to invoke)
